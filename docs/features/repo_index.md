# Index

## gitfs

gitfs has no repository index. There is no manifest file listing available formulas, versions, or metadata. Discovery of available content is done by direct file path lookup within the Git tree. Version information is implicit in branch and tag names (e.g., `v1.2.0`). The absence of an index means there is no structured way to query available packages, their dependencies, or their checksums programmatically.

## spm

spm uses an `SPM-METADATA` file as its repository index. This file is generated by the `spm create_repo` command and lists available `.spm` packages with their metadata including name, version, release, summary, and formula type. The index is a flat file without rich dependency or checksum information. Clients parse `SPM-METADATA` to discover available packages before downloading. The format is specific to Salt's spm subsystem and is not interoperable with other package management tools.

## salt-bundle

salt-bundle uses an `index.yaml` file as its repository index. The `Index` model in `models/index_models.py` defines the structure: a top-level `apiVersion: v1` field, a `generated` timestamp, and a `packages` map from package name to a list of `IndexEntry` objects. Each `IndexEntry` contains `version`, `url`, `digest` (in `sha256:<hex>` format), `created` timestamp, `keywords`, `maintainers`, `sources`, and a `dependencies` list of `PackageDependency` objects. The `generate_index()` function in `repository.py` builds this index by scanning a directory for `.tgz` files, extracting metadata from each archive, and computing SHA256 digests. The index is saved with `dump_yaml()` and fetched by clients via `fetch_index()`, which supports both HTTP and local file paths. Versions within each package entry are sorted latest-first.
