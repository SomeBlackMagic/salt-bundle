# Repo Type

## gitfs

gitfs uses Git repositories as its sole repository type. Each remote is defined by a Git URL (SSH, HTTPS, or local path using the `file://` scheme) in the `gitfs_remotes` list in the Salt master configuration. The master clones each remote into its cache and can track multiple branches, tags, and specific commits. There is no separate concept of a package registry or index — the repository is the Git remote itself, and file addressing is by path within the tree.

## spm

spm uses its own repository format — an spm repo — which is an HTTP server or local directory containing `.spm` package archives and an `SPM-METADATA` index file. The index is generated by running `spm create_repo` on a directory of `.spm` files. Repository URLs are configured in `/etc/salt/spm.repos.d/` as `.repo` files. spm supports both HTTP/HTTPS repositories and local filesystem repositories. There is no support for Git-backed repositories or content-addressable storage.

## salt-bundle

salt-bundle supports two repository types, defined by the `type` field in `RepositoryConfig` (`models/config_models.py`): `"remote"` (default) and `"path"`. A `remote` repository is an HTTP/HTTPS server or a `file://` path serving `.tgz` packages and an `index.yaml` manifest. A `path` repository is a local formula directory that is symlinked directly into `vendor/` without packaging — the `symlink_path_package_to_vendor()` function in `vendor.py` creates the symlink, and `fetch_index_from_path_repo()` in `repository.py` builds a synthetic index from the formula's `.saltbundle.yaml` metadata. Repositories are configured per-project in `.salt-dependencies.yaml` or globally in `~/.config/salt-bundle/config.yaml`.
